## Overview

This document is the formal pre-launch release checklist for InternSwipe v1.0. Every item below must be verified and checked off before the v1.0 Git tag is created on the main branch. Brandon is responsible for signing off on the complete checklist, and no release may proceed without his explicit approval. Each item is written as a verifiable statement that can be confirmed with a specific test or inspection.

---

## 1. Code quality

- [ ] The ESLint linter passes on the entire codebase with zero errors and zero warnings.
- [ ] The TypeScript compiler (`tsc --noEmit`) passes with zero errors across all source files.
- [ ] All Vitest unit tests pass without failures or skipped tests.
- [ ] All Playwright end-to-end tests pass without failures or flaky results.
- [ ] No `console.log` statements remain in production code (development-only logging utilities are acceptable if they are gated by environment checks).
- [ ] No `TODO`, `FIXME`, or `HACK` comments remain in critical code paths (non-critical paths may retain them if they are tracked as GitHub Issues).
- [ ] All pull requests have been reviewed and merged, and no stale feature branches remain open.

## 2. Security

- [x] Signed URLs for resume file access are working and expire correctly after 15 minutes. — **done** (`GET /api/resume/signed-url` with 15-min expiry)
- [x] No resume file is accessible without a valid signed URL generated by an authenticated session. — **done** (endpoint requires auth via `requireAuth()`)
- [x] Auth guard middleware is active on all protected API routes and returns a 401 status code for unauthenticated requests. — **done** (centralized `requireAuth()` on all 6 protected routes)
- [x] Rate limiting is enforced on the `POST /api/apply` endpoint at 10 requests per minute per user, and requests exceeding the limit receive a 429 status code. — **done** (sliding-window `RateLimiter`, 429 with `Retry-After`)
- [ ] No secrets, API keys, or credentials are committed to the repository (verified by searching the Git history for common secret patterns). — **done** (credentials use placeholders in .env.local.example)
- [ ] The `.env.local` file is listed in `.gitignore` and is not tracked by Git. — **done**
- [ ] All environment variables containing secrets are documented in the environment setup guide without exposing their actual values. — **done** (documented in environment-setup.md with placeholder values only)

## 3. Data integrity

- [ ] All seven MVP tables are present in the production Supabase instance and match the Prisma schema exactly. — **done** (schema defines all 7 tables: users, profiles, resumes, jobs, swipe_actions, applications, submission_logs)
- [ ] The seed data has been loaded into the production database, and 25 or more jobs are present with correct eligibility flags. — **done** (seed script inserts 25 jobs — 15 ELIGIBLE, 10 NOT_ELIGIBLE)
- [ ] Every eligibility flag has been manually verified against the original job posting to confirm accuracy.
- [ ] The `POST /api/apply` endpoint correctly prevents duplicate submissions for the same user-job pair. — **done** (returns 409 for duplicate APPLIED status)
- [ ] Application records are created for every submission attempt, and SubmissionLog records are created for every attempt including failures and retries. — **done** (apply route creates both records, including for failed eligibility checks)
- [ ] The retry mechanism only allows re-submission when the prior Application status is FAILED and the user explicitly triggers the retry action.
- [ ] All data displayed in the history view (Applied, Failed, Skipped) matches the corresponding records in the database.

## 4. User experience

- [ ] The application is responsive and functional on mobile devices at 375px viewport width.
- [ ] The application is responsive and functional on tablet devices at 768px viewport width.
- [ ] The application is responsive and functional on desktop devices at 1280px viewport width.
- [ ] All empty states are implemented: no jobs matching filters, no applications yet, and no resume uploaded.
- [x] All error messages displayed to the user are written in plain language and do not expose stack traces, internal identifiers, or cryptic error codes. — **done** (all routes use `formatZodError` for validation and `safeErrorMessage` to prevent internal error leakage)
- [ ] Toast notifications display correctly for applied, failed, and ineligible swipe outcomes.
- [ ] Keyboard navigation works on all interactive elements, and every interactive element has a visible focus state.
- [ ] Screen reader labels are present on all interactive elements and are announced correctly.
- [ ] Color contrast ratios meet WCAG AA standards on all text elements and UI components.
- [ ] All placeholder text ("Lorem ipsum," "TODO," or "TBD") has been replaced with production copy.
- [ ] Swipe card animations run at 60 frames per second without dropped frames or visual glitches on a mobile device.

## 5. Launch artifacts

- [x] The README includes complete setup instructions that allow a new team member to run the project locally from scratch. — **done**
- [x] The README includes documentation of all required environment variables with descriptions of their purpose and where to obtain their values. — **done** (5 env vars documented with source locations)
- [x] The README includes an architecture overview that describes the major components of the system and how they interact. — **done** (file tree, request lifecycle, database schema, security table)
- [x] The handoff document has been written and committed, explaining how to deploy the application, how to add new jobs to the database, and how to extend the system with new features. — **done** (`docs/HANDOFF.md`)
- [x] Release notes have been written and committed, summarizing the features delivered, known limitations, and suggested next steps. — **done** (`docs/RELEASE-NOTES-v1.0.md`)
- [ ] The v1.0 Git tag has been created on the main branch after all checks above have passed.
- [ ] The production URL on Vercel is confirmed live and serving the application without errors.
- [ ] The production deployment has been tested by completing a full user journey: sign-up, profile setup, resume upload, swipe, apply, and history review.
